### ¿Por qué fue una buena decisión de arquitectura usar MongoDB para los platos en lugar de seguir usando SQL?

Es ideal utilizar una base de datos NoSQL para manejar colecciones como **'Platos'** por que en esta clase de base de datos, se permite la creación de colecciones flexibles y permite el uso de datos anidados. En la practica se utilizan distintas categorias o tipo de información como Las bebidas que pueden contener alcohol y su volumen se calcula en ml, como en el  campo `es_alcoholica` (boolean) y `tamano_ml` (int) de las bebidas o los casos de platos fuertes que tienen un campo de  `alergenos` el cual se definio como un array de strings. Así mismo en un futuro se puede incluir otro tipo de alimentos que podrían tener distintos atributos que se deban almacenar. Por tanto Mongo es ideal para manejar 'Platos' por que ofrece **flexibilidad** para datos heterogéneos (Platos tiene variedad de recetas que tienen atributos distintos), simplifica el manejo de **campos opcionales y arrays**, ideal para este caso donde los datos que se almacenan varían entre registros.

Si se usara el modelo SQL, la tabla 'Platos', tendría  muchos campos en una misma tabla que solo funcionarían para registros especificos, lo que implica que abrían muchas columnas opcionales es decir gran parte de lo regitros con valores nulos. Tambien incluiría crear nuevas tablas para tratar de normalizar el modelo. Incrementando el costo de mantenimiento y consultas al tratar de hacer consultas con JOINs.
